# Chore Robot GPR Simulation for NVIDIA Isaac Sim

A structured simulation repository for the Chore Robotics Ground-Penetrating Radar (GPR) perception workflow. It contains Isaac Sim environments for Nova Carter and the Chore Robot, a ROS 2 GPR subsurface node, ground-truth buried-object configuration, and the master perception and motion protocol.

## Main Components

### ROS 2 GPR node

`src/gpr/gpr_subsurface_node.py` simulates a front-mounted subsurface sensing module using Isaac Sim depth and semantic-segmentation streams.

**Subscriptions**

- `/gpr/depth` — `sensor_msgs/Image`, expected encoding `32FC1`
- `/gpr/semantic_segmentation` — `sensor_msgs/Image`
- `/odom` — `nav_msgs/Odometry`

**Publications**

- `/gpr/subsurface_scan` — JSON payload on `std_msgs/String`
- `/gpr/markers` — `visualization_msgs/MarkerArray`

Marker interpretation:

- Green: normal ground
- Yellow: uncertain reading
- Red: buried object or hard-surface anomaly

### Ground-truth manifest

`config/gpr_ground_truth_nova_env.json` defines simulated buried objects for the Nova Carter scene, including a PVC pipe, granite rock, organic root, and copper-sheathed cable. Coordinates are world-frame values specific to the Nova Carter environment.

### Isaac Sim scenes

- `assets/isaac_sim/scenes/nova_carter/` contains the Nova Carter GPR environment.
- `assets/isaac_sim/scenes/chore_robot/` contains the custom Chore Robot GPR environment.

Do not interchange scene-specific coordinates without transforming them into the correct local or world frame.

## Requirements

- NVIDIA Isaac Sim
- ROS 2 Humble or the ROS 2 distribution configured with Isaac Sim
- Python 3
- `rclpy`
- `numpy`
- `cv_bridge`
- `sensor_msgs`
- `std_msgs`
- `nav_msgs`
- `visualization_msgs`

Example ROS 2 package installation on Ubuntu with Humble:

```bash
sudo apt update
sudo apt install -y \
  ros-humble-cv-bridge \
  ros-humble-sensor-msgs \
  ros-humble-std-msgs \
  ros-humble-nav-msgs \
  ros-humble-visualization-msgs \
  python3-numpy
```

## Run Locally in a ROS 2 Environment

```bash
cd chore-robot-gpr-isaac-sim
source /opt/ros/humble/setup.bash
./scripts/run_gpr_node.sh
```

The launcher automatically sets:

```bash
GPR_MANIFEST_PATH=config/gpr_ground_truth_nova_env.json
```

## Run in the Existing Docker Container

From the host:

```bash
docker cp src/gpr/gpr_subsurface_node.py chore_stable:/perception/
docker cp config/gpr_ground_truth_nova_env.json chore_stable:/perception/
```

Then enter the container and run:

```bash
docker exec -it chore_stable bash
source /opt/ros/humble/setup.bash
export GPR_MANIFEST_PATH=/perception/gpr_ground_truth_nova_env.json
python3 /perception/gpr_subsurface_node.py
```

## Verify ROS 2 Topics

```bash
ros2 topic list | grep gpr
ros2 topic echo /gpr/subsurface_scan
ros2 topic hz /gpr/subsurface_scan
ros2 topic echo /gpr/markers
```

## Isaac Sim Workflow

1. Open the relevant `.usda` scene in Isaac Sim.
2. Enable the ROS 2 bridge extension.
3. Confirm that the GPR depth and semantic-segmentation render products publish to the expected topics.
4. Start simulation playback.
5. Launch the ROS 2 GPR node.
6. Add `/gpr/markers` as a `MarkerArray` display in RViz 2.
7. Set the RViz fixed frame to the frame used by the active TF tree, normally `map` for this node.

## Important Implementation Note

The semantic-segmentation ID-to-label mapping in `gpr_subsurface_node.py` contains a placeholder lookup. Before production use, confirm the runtime semantic IDs generated by Isaac Sim and map them to:

- `gpr_pipe`
- `gpr_rock`
- `gpr_root`
- `gpr_cable`

Without this one-time mapping, the node may publish normal-ground readings even when a buried-object marker is visible.

## Project Status

Current scope is a simulation and demonstration implementation. The GPR output is generated from simulated depth, semantic segmentation, configured buried-object metadata, and sensor-noise models; it is not a physics-accurate electromagnetic GPR solver.
